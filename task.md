Техническое задание:
1. Стек технологий (обязательные требования)
	Язык программирования: Python 3.11 или выше.
	Фреймворк для парсинга: На выбор кандидата (Scrapy, BeautifulSoup + Requests, Selenium, Playwright и т.п.) с обязательным обоснованием выбора в комментариях к коду или в README.
	База данных: Реляционная СУБД (PostgreSQL, MySQL или SQLite – для простоты тестирования). Допускается использование SQLite в целях упрощения, но приложение должно быть готово к переключению на внешнюю БД через конфигурацию.
	ORM: SQLAlchemy (версия 2.0 рекомендуется) для взаимодействия с БД.
	Контейнеризация: Docker и Docker Compose для сборки и запуска всего окружения (приложение + БД).
	Дополнительно: Использование asyncio или aiohttp приветствуется, но не обязательно.
2. Функциональные требования
Разрабатываемая программа должна:
1.	Принимать на вход список идентификаторов (например, ИНН) в количестве от 100 и более. Способ передачи данных:
o	Через аргумент в docker compose из файла (.xlsx).
2.	Для каждого идентификатора выполнять парсинг целевого сайта согласно алгоритму, описанному в разделе «Задание». Алгоритм может быть представлен в виде текста и/или графической схемы (прилагается отдельно).
3.	Извлекать необходимые данные (перечень полей будет указан в задании) и приводить их к корректному типу.
4.	Сохранять результаты в базу данных:
o	Спроектировать схему БД (таблицы, связи) самостоятельно, исходя из структуры извлекаемых данных.
o	Использовать SQLAlchemy для описания моделей и выполнения запросов.
o	Обеспечить уникальность записей (например, избегать дублирования по ИНН и дате парсинга).
5.	Обрабатывать ошибки:
o	Сетевые сбои (таймауты, недоступность сервера) – реализовать повторные попытки с экспоненциальной задержкой.
o	Отсутствие данных по идентификатору (например, ИНН не найден) – логировать и продолжать работу.
o	Блокировки со стороны сайта (CAPTCHA, блокировка по IP) – предусмотреть паузы, смену User-Agent, использование прокси (опционально, можно заглушкой).
6.	Массовый ввод:
o	При обработке 100+ идентификаторов программа не должна «упасть» и должна использовать разумные ограничения по нагрузке (задержки между запросами, конкурентность не более N потоков).
o	Желательно реализовать возможность докачки (resume) при прерывании.
7.	Логирование:
o	Записывать в лог-файл (и/или в stdout) этапы работы, ошибки, предупреждения.
o	Формат логов – информативный (время, уровень, сообщение).
3. Нефункциональные требования
	Читаемость кода:
o	Соблюдение PEP 8, осмысленные имена переменных и функций.
o	Комментарии к сложным участкам, документация функций (docstrings).
o	Структурированность проекта (разделение на модули: парсер, модели, работа с БД, конфигурация, утилиты).
	Документация:
o	Файл README.md с описанием:
	назначения программы;
	стека;
	инструкции по установке и запуску (через Docker и локально);
	примера использования;
	описания конфигурации.
	Контейнеризация:
o	Наличие Dockerfile, который собирает образ приложения.
o	Наличие docker-compose.yml, который поднимает сервис парсера и необходимую БД (например, PostgreSQL).
o	Все зависимости должны фиксироваться в requirements.txt или pyproject.toml.
	Обработка ошибок:
o	Программа не должна завершаться аварийно при возникновении исключений, связанных с парсингом отдельных элементов.
o	Должна быть предусмотрена очистка ресурсов (закрытие соединений с БД, сессий).
